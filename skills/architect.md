# Architect: System Architecture Design Command

Transform PRD requirements into technical architecture that answers **HOW to build** the system.

## Critical Principles

**Architecture = HOW (Technical Design)**:

- PRD defines WHAT and WHY
- Architecture defines HOW to implement
- Includes concrete examples, code interfaces, diagrams
- Technical decisions with justifications

**Reference-Driven Design**:

- Study `docs/ARCHITECTURE.md` for format
- Follow existing codebase patterns
- Reuse proven solutions from [shared-lib]
- Document deviations with rationale

**Practical and Concrete**:

- Include code examples (interfaces, flows, configs)
- ASCII diagrams for visual clarity
- Swimlane diagrams for complex flows
- Specific technology choices with justifications

## Workflow

### Phase 1: Research & Context

**1.1 Read PRD**: Understand functional/non-functional requirements, MVP scope

**1.2 Verify Actual Patterns** (Code > Documentation):

```bash
# Check actual technology choices
# Go example
grep -r "gorm\|database/sql\|pgx" apps/*/go.mod
ls apps/*/generate.go apps/*/oapi-config.yaml

# Python example
grep -r "sqlalchemy\|psycopg\|asyncpg\|django\|fastapi" apps/*/pyproject.toml
ls apps/*/pyproject.toml apps/*/openapi.yml

# Primary references (in order)
# - [reference-app]/ (simple stateless services)
# - Other existing services (additional complexity)
# - Complex services (may not follow latest patterns)
```

**Architectural Principles**:

1. **Single Source of Truth**: Single config file (`config.go` / `config.py`), DATABASE_URL, document data flows in ARCHITECTURE.md
2. **Use Standard Tools**: Go (`go:generate`, `samber/lo`, OAPI validation), Python (`scripts/generate_*.py` or `make generate`, standard library/tooling, OAPI validation)
3. **Organize by Concern**: models/, repository/, co-locate related code
4. **Simplify Everything**: Fewer config sources, connection strings > component variables
5. **Document Non-Code Only**: Data flows and integrations (YES), code structure (NO - code shows this)

**1.3 Clarify Scope**: Ask user about depth, focus areas, constraints, existing code integration

### Phase 2: Document Structure

Follow this structure (based on project ARCHITECTURE.md):

1. **System Overview** - Purpose, responsibilities, context diagram
2. **Architecture Principles** - Design principles, standards, SOLID
3. **API Design** (if applicable) - Endpoints, standards, OpenAPI
4. **Component Architecture** - Layers, directory structure, responsibilities
5. **Data Flow** - Request/response flow with swimlane diagrams
6. **Data Architecture** (if applicable) - Domain models, storage, repositories
7. **External Integrations** (if applicable) - Provider system, clients
8. **Security Architecture** (if applicable) - Auth, encryption, certificates
9. **Configuration** - Environment variables, secrets
10. **Testing Strategy** - Unit, component, integration, frontend (Vitest + Playwright)
11. **Deployment** - Infrastructure, CI/CD
12. **Observability Design** - Logging, metrics, tracing, alerts, SLOs
13. **Success Metrics** - Performance targets
14. **References** - Links to PRD, specs, standards, project README

### Phase 3: System Overview

**Purpose & Responsibilities**: What system does, key responsibilities (4-6 bullets), out of scope

**Context Diagram**: ASCII showing clients, system, external dependencies, protocols

### Phase 4: Architecture Principles

**Design Principles**: Framework consistency, type safety, interface-based design, single responsibility, testability, dependency injection, schema-driven (if OpenAPI)

**Standards Compliance** (if applicable): ISO (countries, currencies, dates, IBAN), RFC (JWT, OAuth), industry standards, data formats (JSON, UTF-8, snake_case)

**SOLID Principles**: Map to your design (SRP, OCP, LSP, ISP, DIP)

### Phase 5: Component Architecture

**5.1 Layered Architecture Diagram**: API Layer → Service Layer → Repository/Client

**5.2 Directory Structure** (OpenAPI-first pattern with oapi-codegen/Python OpenAPI generators):

**Go example**:

```
apps/[service]/
├── cmd/api/main.go
├── app.go
├── PRD.md
├── ARCHITECTURE.md
├── openapi.yml                  # SOURCE OF TRUTH (to be created)
├── internal/
│   ├── config.go
│   ├── domain/                  # Domain entities
│   ├── api/
│   │   ├── gen/                 # GENERATED by oapi-codegen (DO NOT EDIT)
│   │   ├── handlers/            # Handler implementations
│   │   └── router.go            # Route registration
│   ├── services/                # Business logic
│   └── store/                   # Data access
├── test/component/              # HTTP endpoint tests
├── go.mod
└── README.md
```

**Python example**:

```
apps/[service]/
├── app/
│   ├── main.py
│   ├── config.py
│   ├── domain/                  # Domain entities
│   ├── api/
│   │   ├── gen/                 # GENERATED by openapi-python-client/datamodel-code-generator (DO NOT EDIT)
│   │   ├── routers/             # Router implementations
│   │   └── dependencies.py      # Dependency wiring
│   ├── services/                # Business logic
│   └── repository/              # Data access
├── PRD.md
├── ARCHITECTURE.md
├── openapi.yml                  # SOURCE OF TRUTH (to be created)
├── tests/component/             # HTTP endpoint tests
├── pyproject.toml
└── README.md
```

**Vue frontend example**:

```
frontend/  (或 apps/[service]-web/)
├── src/
│   ├── main.ts
│   ├── App.vue
│   ├── components/          # 可复用组件
│   ├── views/               # 页面级组件（对应路由）
│   ├── composables/         # 组合式函数 (业务逻辑)
│   ├── stores/              # Pinia 状态管理
│   ├── router/              # Vue Router 路由配置
│   ├── api/                 # API 调用层
│   ├── types/               # TypeScript 类型定义
│   └── assets/              # 静态资源
├── tests/
│   ├── unit/                # Vitest 组件测试
│   │   ├── components/
│   │   └── composables/
│   ├── e2e/                 # Playwright E2E 测试
│   └── setup.ts             # 测试全局配置
├── vite.config.ts
├── vitest.config.ts
├── playwright.config.ts
├── package.json
└── tsconfig.json
```

**5.3 Component Responsibilities**: For each layer, document purpose, responsibilities, key files, what's from shared-lib/oapi-codegen (Go), shared-lib/Python OpenAPI generators (Python), and app-specific code

## Document Style Guidelines

- **Concise**: Scannable structure, clear headings
- **Visual**: ASCII diagrams for architecture, swimlanes for flows
- **Practical**: Code interfaces, configs, specific examples
- **Justified**: Explain technical decisions

## Code Example Policy

✅ **DO**: Interface signatures, config structs, key type definitions (<10 lines)
❌ **DON'T**: Full implementations, complete files, >10 line code blocks

## Output Location

Save to: `docs/ARCHITECTURE.md` (or `apps/[service]/docs/ARCHITECTURE.md`)

**Header**: Copy from `~/.claude/templates/architecture-header.md` and customize service name, date

## Remember

✅ **DO**:

- Reuse shared-lib components (server, router, logger, config, middleware)
- Follow [reference-app]/ for proper lib usage pattern
- Use OpenAPI-first pattern (Go: openapi.yml → oapi-codegen → implement gen.ServerInterface; Python: openapi.yml → datamodel-code-generator/openapi-python-client → implement FastAPI routers)
- Include minimal code examples (interfaces, configs, <10 lines)
- Use ASCII diagrams for visual clarity
- Be specific (name actual files, components, technologies)
- Justify choices (explain why, not just what)
- Reference standards ([infrastructure-repo]/README.md, schemas/, PRD)
- Follow [framework stack per project README]
- Document that openapi.yml is source of truth (to be created)
- Mark internal/api/gen/ as GENERATED
- Include complete Observability Design section

❌ **DON'T**:

- Reinvent shared-lib components (custom HTTP server, health endpoints, logger, config, middleware)
- Hand-write OpenAPI-generated code (request/response structs, route wiring)
- Edit files in internal/api/gen/ (they are GENERATED)
- Include full OpenAPI spec in ARCHITECTURE.md (it's an output, not architecture)
- Include code examples >10 lines

**Now create a detailed, practical ARCHITECTURE.md following the established format!**
